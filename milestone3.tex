\section{Introduction}
This Design Report outlines the technical architecture and detailed system design for the Student Productivity Platform. Following the requirements established in the SRS, this document serves as a blueprint for the implementation phase. It covers the structural modeling (Class, Component) and behavioral modeling (Sequence, Activity) of the core modules: Dashboard/LMS, Task/Gamification, and Collaboration/Wellness.

% ============================================
% 2. DESIGN ASSUMPTIONS AND CONSTRAINTS
% ============================================
\section{Design Assumptions and Constraints}

\subsection{Design Assumptions}
\begin{itemize}
    \item \textbf{API Reliability:} It is assumed that the University's LMS provides a consistent RESTful API interface with high availability.
    \item \textbf{Data Mapping:} We assume a compatible mapping between LMS assignment categories and internal task structures.
\end{itemize}

\subsection{Design Constraints}
\begin{itemize}
    \item \textbf{Privacy:} Personal wellness and stress logs must be encrypted to ensure strict data protection.
    \item \textbf{Performance:} The system must sync and display LMS data on the dashboard within 2 seconds of the user request.
\end{itemize}

% ============================================
% 3. KEY DESIGN DECISIONS
% ============================================
\section{Key Design Decisions}
\begin{itemize}
    \item \textbf{Polling Strategy:} A 15-minute polling interval was selected to balance data freshness with mobile device battery optimization.
    \item \textbf{Modular Design:} Each module (LMS, Tasks, Wellness) is architected to be independent to facilitate parallel development and maintenance.
\end{itemize}
